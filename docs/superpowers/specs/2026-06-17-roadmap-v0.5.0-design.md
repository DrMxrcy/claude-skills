# roadmap v0.5.0 — "control & visibility"

**Status:** Approved (design)
**Date:** 2026-06-17
**Skill:** `skills/roadmap`
**Target version:** `0.4.0` → `0.5.0`

## Summary

Six backward-compatible additions to the roadmap skill, grouped into three themes
the user selected: **drift-proofing**, **visibility/sharing**, and **planning power**,
plus a project-level **rules refresh** command surfaced during design.

All changes are additive. Existing `config.json` and plan files keep working: every
new field is optional and read tolerantly (absent → rendered as "—" / treated as
empty). No migration step is required.

## Motivation

- The skill's core promise is an anti-drift tracking layer, but progress only updates
  when `check` runs. Work done outside the commands silently leaves the dashboard
  behind (the user hit this in practice). `/roadmap:catchup` (v0.4.0) is the recovery
  path; v0.5.0 makes drift **visible and self-correcting** rather than relying on the
  model remembering a rule.
- There is no stakeholder-facing progress view beyond the raw dashboard, and no sense
  of pace.
- Items have no ordering relationship, so `next`/`build` can pick an item whose
  prerequisites aren't done.
- The Idea Incubator is free-form text with no bridge into the tracker.
- Updating the skill globally (`curl … | bash --global`) deliberately skips
  `CLAUDE.md`, so existing projects never receive updated rules. There is no clean,
  obvious "refresh this project's rules" command.

## Scope (six items)

### 1. SessionStart orientation (inject into context)

- New hook `hooks/roadmap-orient.sh` runs `roadmap.py orient`.
- `orient` emits SessionStart `additionalContext` JSON containing: current version,
  progress (`x/y steps`, `n/m items`), and the **next unchecked item** (id, title,
  type). This injects into Claude's context so every session auto-orients instead of
  depending on the model remembering "check status first."
- Installer wires it idempotently (same JSON-merge pattern as the Stop hook), with an
  opt-out flag `--no-orient`.
- If there is no `.roadmap/` in the project, `orient` is a no-op (emits nothing /
  exit 0) so the hook is harmless in non-roadmap repos.

### 2. Drift nudge (Stop hook)

- `config.json` gains `last_seen_sha` (string, optional).
- New CLI `drift-check`: if the repo is a git repo and `git rev-parse HEAD` differs
  from `last_seen_sha` **and** the current version still has unchecked steps, print
  `⚠ N commit(s) since last check-off — run /roadmap:catchup?` to stdout and exit 0.
  Otherwise silent.
- `check` updates `last_seen_sha` to current HEAD after a successful check-off (so the
  nudge clears once you reconcile).
- The existing Stop hook (`roadmap-sync.sh`) calls `drift-check` after `sync`. Never
  auto-checks anything — detection only.
- Non-git repos / no `.roadmap/`: silent no-op.

### 3. `/roadmap:report` → file

- New CLI `report` writes `docs/roadmap-report.md` (regenerated each run, fully
  managed/overwritten).
- Contents:
  - Title + generated-at date + current version.
  - Per-version section with a progress bar and item list grouped
    ✅ shipped (released) / 🔄 in progress / ⬜ planned.
  - Velocity summary (from item 4).
- New slash command `commands/roadmap/report.md`.

### 4. Velocity (record dates + show)

- Items gain `completed_at` (ISO date), set when the item's **last** step is checked.
- Versions gain `released_at` (ISO date), set on `release`.
- `report` shows items-per-version and recent pace (e.g. items completed in the last
  N days, average items per released version).
- Configs written before v0.5.0 lack these fields → rendered as "—"; no backfill
  required (the existing `/roadmap:changelog` git-backfill remains the path for
  historical data).

### 5. Dependencies (warn + skip)

- Plan frontmatter gains optional `depends_on: [ids]` (list of plan ids).
- `next`: skips items whose dependencies are not 100 % complete; reports what it
  skipped and why.
- `build` (single item): if a dependency is incomplete, **warn** and continue;
  `--force` suppresses the warning. (Not a hard block — keeps the tool non-rigid.)
- `status` and `report` show blocked items and their blockers.
- `plan` can record `depends_on` when creating an item.
- Guard against cycles: `depends_on` validation rejects self-reference and reports a
  cycle rather than looping.

### 6. `/roadmap:promote` and `/roadmap:upgrade`

- **promote** (`commands/roadmap/promote.md`): reads the Idea Incubator section of
  `ROADMAP.md` (free-form text outside the managed markers), the user picks an entry,
  it runs the normal `plan` flow into a tracked item, then removes that line from the
  incubator. Only the chosen line is removed; the rest of the incubator is preserved.
- **upgrade** (CLI `upgrade`, `commands/roadmap/upgrade.md`): a **project-level**
  command — run inside a project to re-inject the current `RULES_BLOCK` into that
  project's `CLAUDE.md` (idempotent, between the `roadmap:rules` markers), report the
  old → new skill version, and resync `ROADMAP.md`. This is the post-update
  muscle-memory command: after `curl … | bash --global` updates the skill, run
  `/roadmap:upgrade` in each project to pull the new rules. Does **not** require
  re-running the installer.

## Cross-cutting requirements

- `RULES_BLOCK` must stay byte-for-byte identical across `roadmap.py`, `install.sh`,
  and `skills/roadmap/example/CLAUDE.md`. v0.5.0 adds lines referencing the new
  `promote`/`upgrade` workflow; update all three together and verify parity.
- `VERSION` → `0.5.0`.
- Update `skills/roadmap/README.md` (command table + new sections) and the top-level
  update docs to point at `/roadmap:upgrade` after a global update.
- Installer: wire the new SessionStart orient hook; document `--no-orient`.

## Testing

- Unit + e2e tests for every new CLI subcommand: `orient`, `drift-check`, `report`,
  `upgrade`, and the `depends_on` behavior in `next`/`build`, plus `completed_at` /
  `released_at` recording.
- Hook tests: orient emits valid SessionStart JSON; drift-check nudges only when HEAD
  moved AND items incomplete; both are silent no-ops without `.roadmap/`.
- Backward-compat test: a pre-v0.5.0 `config.json` (no new fields) loads and renders
  without error.
- Parity test: `RULES_BLOCK` identical across the three files (extend existing check).
- Run with `python3.11 -m pytest` (the environment's `python3` is broken;
  `GIT_CONFIG_GLOBAL=/dev/null` for hermetic git in repo fixtures).

## Non-goals (deferred)

- GitHub Issues/Projects sync and a CI/test gate on `release` (theme D) — not in this
  milestone.
- File-aware drift matching (mapping changed files to specific steps) — the
  commits-moved + incomplete heuristic is intentionally simple.
- Hard-blocking dependencies — warn+skip only for now.
- Velocity backfill of historical completion dates beyond what changelog backfill
  already covers.

## Anti-drift state model (unchanged)

State still lives in the target project: `ROADMAP.md` (rendered view between
`roadmap:auto` markers), `.roadmap/config.json` (ids, version, registry — now also
`last_seen_sha`, `released_at`, and per-item `completed_at`), `.roadmap/plans/NNN-*.md`
(checklist source of truth — now also optional `depends_on`), `CHANGELOG.md`, and the
new managed `docs/roadmap-report.md`.
